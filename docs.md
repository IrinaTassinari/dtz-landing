# Документация по лендингу DTZ

## Как добавить новый язык

Текущая локализация хранится в JSON-файлах в `src/locales` и подключается через `src/lib/i18n.ts`.

Порядок действий:

1. Создать новый файл локали в `src/locales`, например `tr.json`.
2. Скопировать структуру из существующего файла, например `en.json`, и перевести все значения.
3. Подключить новый JSON в `src/lib/i18n.ts`:
   - добавить `import tr from "@/locales/tr.json";`
   - добавить `tr` в объект `translations`
   - добавить язык в `supportedLanguages`, например `{ code: "tr", label: "TR", name: "Turkce" }`
4. Проверить, что язык открывается по маршруту `/tr`.
5. Пройтись по странице и проверить переносы строк, длину заголовков и карточек.
6. Запустить `npm run lint`.

Оценка времени:

- техническое подключение языка: 10-15 минут
- перенос переводов всех ключей: 20-60 минут
- визуальная проверка и мелкие правки: 15-30 минут
- всего: обычно 45-90 минут на один язык

## Сколько стоит владение

Для текущего проекта стоимость низкая, потому что это SEO-лендинг на Next.js без тяжелого backend.

Примерный состав затрат:

- хостинг: Vercel Hobby бесплатно, Vercel Pro около 20 EUR/мес
- домен: обычно около 8-15 EUR в год, то есть примерно 1 EUR в месяц
- CDN: уже входит в Vercel
- аналитика: можно использовать Vercel Analytics или простой бесплатный сервис
- поддержка: 1-4 часа в месяц на правки текстов, ссылок, секций, локалей и UI

Реалистичная оценка:

- минимально: около 1 EUR/мес плюс домен, если оставаться на бесплатном тарифе
- рабочий проект: около 20-25 EUR/мес плюс домен
- если закладывать человека на сопровождение, добавляется стоимость его часов

## Сколько будет стоить 10K посетителей

Для такого лендинга 10 000 посетителей в месяц не должны быть дорогими.

При текущем стеке:

- трафик и CDN, скорее всего, поместятся в бесплатные лимиты Vercel Hobby
- аналитика тоже обычно укладывается в бесплатный объем
- дополнительных расходов на backend при обычной посещаемости не ожидается

Оценка:

- на Hobby: скорее всего 0 EUR дополнительной инфраструктурной стоимости
- на Pro: базовый тариф около 20 EUR/мес уже покрывает такой объем

Стоимость начнет заметно расти, если:

- добавить видео или тяжелые изображения
- сделать много серверной логики
- подключить платные внешние сервисы

## Как добавить новую секцию

Сейчас страница собрана в `src/app/[lang]/page.tsx`, а тексты лежат в JSON-локалях.

Порядок действий:

1. Добавить новые текстовые ключи во все локали:
   - `src/locales/ru.json`
   - `src/locales/de.json`
   - `src/locales/en.json`
   - `src/locales/ar.json`
2. Добавить новую `<section>` в `src/app/[lang]/page.tsx`.
3. Использовать существующие CSS-переменные темы, например:
   - `var(--surface)`
   - `var(--line)`
   - `var(--accent-strong)`
4. Проверить логичную иерархию заголовков:
   - секция обычно `h2`
   - внутренние подразделы обычно `h3`
5. Проверить светлую и темную тему.
6. Запустить `npm run lint`.

Оценка времени:

- простая текстовая секция: 20-40 минут
- секция с карточками и сложным layout: 45-120 минут
- если нужны дополнительные локали, mobile-polish и dark mode polish: до 2-3 часов

## Какой стек выбран и почему

Текущий стек:

- Next.js 16
- React 19
- TypeScript 5
- Tailwind CSS 4
- App Router
- `next/font`
- JSON-локали

Почему он подходит:

- Next.js хорошо подходит для SEO-лендинга и дает удобный путь роста в полноценный продукт
- React удобен для повторяющихся секций и UI-компонентов
- TypeScript снижает риск ошибок при росте количества секций и языков
- Tailwind ускоряет сборку адаптивного интерфейса
- JSON-локали позволяют быстро добавлять новые языки без отдельного i18n-сервиса
- CSS variables и theme toggle позволяют поддерживать светлую и темную тему без дублирования верстки

## Почему JSON, а не i18n-библиотека

Для этого проекта выбраны JSON-локали вместо отдельной i18n-библиотеки, потому что для текущего масштаба это более простое и рациональное решение.

Почему это оправдано:

- проект представляет собой в первую очередь лендинг, а не большое многоэкранное приложение
- большая часть контента статическая и заранее известна
- тексты не требуют сложной runtime-обработки
- не нужны ICU-formatting, plural rules, message parsing и сложная интерполяция строк
- не нужен отдельный provider, middleware и дополнительная инфраструктура локализации
- новый язык добавляется прозрачно: новый JSON-файл плюс одна запись в `src/lib/i18n.ts`
- уменьшается количество зависимостей и техническая сложность проекта
- ниже риск регрессий и конфликтов между server/client-частями
- проще передавать проект другому разработчику, редактору или контент-менеджеру

Практический плюс такого подхода:

- переводы лежат в явном виде в `src/locales`
- логика выбора языка сосредоточена в одном месте, в `src/lib/i18n.ts`
- языковая версия страницы определяется маршрутом `src/app/[lang]/page.tsx`
- структура понятна даже без знания специализированной i18n-библиотеки

Когда отдельная i18n-библиотека была бы оправдана:

- если в проекте станет много динамических переводов
- если понадобятся склонения, pluralization и форматируемые переменные в строках
- если появятся вложенные namespaces и более сложная организация переводов
- если потребуется развитая server/client i18n-инфраструктура
- если лендинг вырастет в полноценный продукт с большим количеством экранов и пользовательских сценариев

Итоговое решение:

На текущем этапе JSON-локали дают лучший баланс между скоростью разработки, простотой поддержки и достаточной гибкостью. Если проект вырастет по сложности, переход на полноценную i18n-библиотеку останется возможным без смены общей структуры маршрутов.

Где это применяется:

- React 19: компоненты в `src/app/[lang]/page.tsx`, `src/components/theme-toggle.tsx`, `src/components/language-switcher.tsx`
- App Router: маршруты в `src/app/page.tsx`, `src/app/[lang]/page.tsx`, общий layout в `src/app/layout.tsx`
- `next/font`: подключение шрифтов в `src/app/layout.tsx`

Текущее ограничение архитектуры:

- почти весь лендинг собран в одном крупном файле `src/app/[lang]/page.tsx`

Если проект продолжит расти, следующий шаг:

- вынести секции в отдельные компоненты
- при необходимости разбить локали по секциям
